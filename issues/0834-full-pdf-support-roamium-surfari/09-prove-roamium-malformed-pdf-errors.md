# Experiment 9: Prove Roamium Malformed PDF Errors

## Description

Experiments 2 through 8 have proven current Roamium support for baseline PDF
rendering, keyboard navigation, page selector behavior, links, find/search,
document restriction behavior, and password-protected PDFs. Another required
Roamium row from the Issue 834 matrix remains unproven: malformed/error PDFs.

This experiment should prove how Roamium handles invalid or truncated PDF
documents through the real TermSurf/Roamium PDF path. The goal is not to make
Chromium's error UI match any other engine. The goal is to prove that malformed
PDFs fail safely and visibly: Roamium should not crash, hang, leak stale loaded
PDF state, or leave the TUI/GUI without a deterministic failure observable.

## Changes

1. Create deterministic malformed PDF fixtures inside the harness log directory.

   Use at least:

   - a non-empty file with a PDF header but truncated body;
   - a file served as `application/pdf` whose bytes are not a PDF;
   - optionally, a zero-byte PDF response if it exposes a distinct stable error
     path.

   Keep fixtures generated by the harness rather than adding binary fixtures to
   the repo unless the result proves a static fixture is necessary.

2. Add a focused malformed PDF probe.

   Create `scripts/test-issue-834-pdf-errors.py` and, if useful,
   `scripts/probe-pdf-error.mjs`. Reuse the TermSurf socket/protobuf and
   DevTools patterns from prior Issue 834 probes.

   The harness should:

   - launch repo-built `chromium/src/out/Default/roamium`;
   - serve malformed fixtures from a local HTTP server;
   - create a Roamium tab through the TermSurf protocol;
   - resize the tab to a stable viewport;
   - discover the DevTools port and any PDF extension child target;
   - observe PDF viewer error state, failed load state, visible error UI, or
     another stable Chromium/PDF observable;
   - detect whether Roamium exits, crashes, hangs, or disconnects;
   - write one summary JSON file at `<log-dir>/pdf-error-summary.json`.

3. Include a valid-PDF control.

   The malformed probes must not be trusted unless the same harness can load a
   known valid PDF. Use `test-html/public/bitcoin.pdf` or a generated minimal
   valid PDF as the control.

4. Prove stale loaded-state cleanup.

   Add a same-process, same-tab transition scenario:

   - load the valid control PDF;
   - capture the successful loaded-plugin/viewer state;
   - navigate the same tab to each malformed fixture;
   - prove the prior successful PDF state is cleared, replaced by a stable error
     observable, or otherwise no longer reported as a normal loaded PDF.

   This is required because a separate valid control only proves the harness can
   load PDFs. It does not prove malformed loads cannot leak stale success state
   from a previous document.

5. Classify the first failing layer.

   If malformed/error handling does not produce a stable observable, classify
   the first failing layer before changing product code:

   - fixture generation;
   - HTTP serving/content-type;
   - Roamium registration;
   - tab creation;
   - DevTools target discovery;
   - PDF viewer error-state discovery;
   - stale success state after malformed load;
   - Roamium/Chromium crash;
   - timeout/hang;
   - evidence collection gap.

6. Fix only a real TermSurf integration bug.

   If Chromium already shows a stable error UI and Roamium stays alive, this
   experiment should remain a harness/evidence experiment. If the probe exposes
   a real crash, hang, stale state, or missing error propagation caused by
   TermSurf integration code, fix only that gap.

   If Chromium source must change, create a fresh branch from the current Issue
   834 Chromium branch, update `chromium/README.md`, build
   `libtermsurf_chromium`, and regenerate the cumulative
   `chromium/patches/issue-834/` archive.

   If non-Chromium product source changes, run the relevant component checks. At
   minimum, Rust changes require `cargo fmt`; Roamium, webtui, Ghostboard, or
   protocol changes require the narrowest relevant build or test command that
   proves the touched component still works.

## Verification

Verification for the completed result is:

Run the probes with fresh log directories, including commands equivalent to:

```bash
python3 scripts/test-issue-834-pdf-errors.py \
  --log-dir logs/issue-834-exp9-errors-control \
  --probe valid-control

python3 scripts/test-issue-834-pdf-errors.py \
  --log-dir logs/issue-834-exp9-errors-malformed \
  --probe malformed-fixtures

python3 scripts/test-issue-834-pdf-errors.py \
  --log-dir logs/issue-834-exp9-errors-stale-state \
  --probe valid-to-malformed-same-tab
```

- valid-PDF control exits 0 and records
  `first_failing_hop = "no-failure-observed"`;
- every malformed fixture records a deterministic outcome;
- malformed PDFs do not crash Roamium;
- malformed PDFs do not report the normal loaded-plugin success state used for
  valid PDFs unless the result explains an equivalent Chromium error state;
- the same-tab valid-to-malformed probe proves successful loaded PDF state does
  not leak after navigation to malformed content;
- the summary records fixture kind, URL, HTTP status/content-type, DevTools
  target discovery, PDF viewer/plugin states, error UI/state evidence,
  process-exit/crash state, timeout state, and first failing hop;
- if a product fix is made, the before/after evidence proves the fix;
- if Chromium source changes, Chromium branch docs, cumulative patch archive,
  and `autoninja -C out/Default libtermsurf_chromium` are updated and passing;
- if Rust source changes, `cargo fmt` is run and the narrowest relevant
  `cargo build`/`cargo test` or `./scripts/build.sh <component>` command passes;
- if Ghostboard source changes, the narrowest relevant `zig build` or
  `macos/build.nu` command passes;
- if only harness code changes, no Chromium, Roamium, Ghostboard, or protocol
  code changes are present;
- any new Node helper passes `node --check`;
- any new Python helper passes
  `PYTHONDONTWRITEBYTECODE=1 python3 -m py_compile`, and `scripts/__pycache__/`
  is removed afterward;
- markdown is formatted with Prettier;
- `git diff --check` passes;
- design review is recorded, all required design-review findings are fixed, the
  design is approved, and the plan commit exists before implementation begins;
- completion review is recorded before the result commit.

## Pass Criteria

This experiment passes if the valid control loads successfully and malformed
PDFs produce stable, safe, non-crashing error outcomes through Roamium with
concrete evidence.

## Partial Criteria

This experiment is partial if the valid control works and at least one malformed
fixture is classified, but another malformed fixture exposes a real product or
automation gap that cannot be safely fixed in this experiment.

## Failure Criteria

This experiment fails if the valid control cannot load, malformed fixtures are
not deterministic, Roamium crashes or hangs without classification, or the
result claims malformed/error support without a stable error observable.

## Design Review

An adversarial Codex subagent reviewed the design with fresh context.

Initial verdict: **Changes Required**.

Required findings:

- The design did not actually prove stale loaded-PDF state could not leak,
  because the valid control was separate from malformed loads. The design now
  requires a same-process, same-tab valid-to-malformed transition scenario.
- The product-code fix path lacked conditional project hygiene checks for
  non-Chromium changes. The design now requires Rust formatting/build or test
  checks and Ghostboard build checks when those areas are touched.

Optional finding:

- The verification section lacked runnable probe commands. The design now lists
  concrete commands and log directories for the control, malformed fixtures, and
  same-tab stale-state probes.

Re-review verdict: **Approved**.
