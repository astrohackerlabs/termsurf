# Experiment 6: Prove Roamium PDF Document Restrictions

## Description

Experiment 5 proved Roamium PDF find/search through TermSurf protocol keyboard
input. The next unproven Roamium PDF rows are document restrictions:

- copy-restricted PDFs;
- save/download-restricted PDFs;
- disabled toolbar states for document restrictions.

These rows should be tested together because they depend on the same core
fixture and viewer state: a PDF whose permission flags deny text extraction and
deny saving/downloading. This experiment should first prove the fixture is valid
and then test current Roamium behavior before changing product code.

## Changes

1. Create or locate deterministic restricted PDF fixtures.

   Prefer a small generated fixture committed under `test-html/public/` only if
   the generated binary is deterministic and small. Otherwise generate it inside
   the harness log directory and record the generation command in the summary.

   The fixture plan should attempt, in order:

   - reuse any existing checked-in restricted PDF fixture if one exists;
   - generate a restricted PDF from a small source PDF with `qpdf` or another
     deterministic local tool;
   - if no valid generator/tool is available, record the fixture-generation
     blocker as the experiment result before touching product code.

   The fixture must have an unrestricted control counterpart so the probe can
   distinguish "copy/save is broken globally" from "restriction enforcement
   works."

2. Add a focused Roamium PDF restriction probe.

   Create `scripts/test-issue-834-pdf-restrictions.py` plus a small DevTools
   helper if useful, likely `scripts/probe-pdf-restrictions.mjs`. Reuse the
   TermSurf socket/protobuf and DevTools patterns from Experiments 3 through 5.

   The harness should:

   - launch `chromium/src/out/Default/roamium` with trace/log output under
     `--log-dir`;
   - serve unrestricted and restricted PDF fixtures from a deterministic local
     HTTP server;
   - create a Roamium tab through the TermSurf protocol;
   - resize the tab to a stable viewport;
   - discover the DevTools port and PDF extension child target;
   - inspect PDF viewer state and toolbar button disabled states;
   - focus the PDF plugin through TermSurf protocol mouse input;
   - attempt text selection/copy through TermSurf protocol keyboard input;
   - attempt save/download through the PDF toolbar or the same user-level path
     used by prior save/download probes;
   - write one summary JSON file at `<log-dir>/pdf-restrictions-summary.json`.

3. Probe unrestricted control behavior first.

   The unrestricted control must prove the harness can observe positive copy and
   save behavior in the same run or in a paired run. If the control cannot copy
   or save, classify the failure as a harness/product regression before making
   claims about document restrictions.

4. Probe restricted behavior.

   The restricted probe should pass only if the evidence shows:

   - copy is blocked or returns no copied document text while the unrestricted
     control copies text;
   - save/download is blocked or the toolbar exposes a disabled state while the
     unrestricted control can save/download;
   - disabled toolbar state is captured for the relevant controls when Chromium
     exposes such state.

   If Chromium enforces a restriction through behavior rather than visible
   disabled state, document that precisely and keep the disabled-state row
   `Partial` until a UI-state observable is found or explicitly accepted.

5. Run the probes.

   Use fresh log directories:

   ```bash
   python3 scripts/test-issue-834-pdf-restrictions.py \
     --log-dir logs/issue-834-exp6-unrestricted-control \
     --probe unrestricted-control
   python3 scripts/test-issue-834-pdf-restrictions.py \
     --log-dir logs/issue-834-exp6-restricted \
     --probe restricted-document
   ```

6. If a probe fails, record the first failing layer.

   Classify the first failing layer before changing product code:

   - fixture generation;
   - fixture validity;
   - protocol mouse/key delivery;
   - clipboard/copy observation;
   - download/save observation;
   - toolbar-state discovery;
   - Chromium PDF restriction plumbing;
   - Roamium integration;
   - evidence collection.

   If the failure is a real TermSurf/Roamium integration gap, fix only that gap
   in this experiment and rerun the relevant probe. If Chromium source must
   change, create a fresh Chromium branch for this experiment and update the
   Chromium branch table and patch archive according to `chromium/AGENTS.md`.

## Verification

Verification for the completed result is:

- restricted and unrestricted fixtures are deterministic or generated by a
  documented deterministic command;
- the unrestricted control proves ordinary copy and save/download remain
  observable;
- the restricted probe uses TermSurf protocol input for copy attempts;
- save/download attempts use the user-visible PDF toolbar or the already-proven
  save/download path from prior PDF probes;
- toolbar disabled-state inspection is recorded with concrete selectors,
  properties, and before/after values;
- the summary records protocol key count, protocol mouse count, Roamium trace
  evidence, Chromium/PDF trace evidence when available, clipboard/download
  evidence, toolbar-state evidence, and first failing hop;
- the result cites command, exit status, summary file, summary status, first
  failing hop, and matrix rows proven or not proven;
- no product code is changed unless the probe exposes a real TermSurf
  integration bug and that fix is explicitly documented in this experiment;
- no Chromium source is changed unless a fresh Chromium branch and patch archive
  are created according to `chromium/AGENTS.md`;
- syntax checks pass for any new Python or Node scripts;
- markdown is formatted with Prettier;
- `git diff --check` passes;
- design review is recorded and the plan commit exists before implementation
  begins;
- completion review is recorded before the result commit.

## Pass Criteria

This experiment passes if Roamium proves all three rows with current evidence:
copy restrictions, save/download restrictions, and disabled toolbar states for
restricted PDFs.

## Partial Criteria

This experiment is partial if it proves at least one restriction row and records
a concrete first failing layer for the remaining row, or if Chromium enforces a
restriction behaviorally but no stable disabled-toolbar observable is found.

## Failure Criteria

This experiment fails if no valid restricted fixture can be produced, if the
unrestricted control does not work, if restriction claims are made without a
control comparison, or if the probe bypasses the TermSurf protocol input path
for copy behavior.
