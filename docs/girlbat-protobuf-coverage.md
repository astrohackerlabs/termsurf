# Girlbat Protobuf Coverage

This matrix tracks Girlbat coverage for every wire message in
`TermSurfMessage.oneof msg` from `proto/termsurf.proto`.

Protocol coverage is not the same thing as full browser or product parity. See
[Girlbat capabilities](girlbat-capabilities.md) for the current user- and
developer-facing limitation report.

The inventory is enforced by:

```bash
python3 scripts/build-girlbat-protobuf-coverage.py --check
```

That check is intentionally inventory-only. It proves that every current wire
message appears exactly once and that the matrix does not mention stale wire
messages. It does not prove Issue 884 is complete. Before Issue 884 can close, a
later strict closure gate must fail if any engine-relevant Girlbat row remains
`Missing`, lacks verified implementation evidence, or lacks an explicit
unsupported/capability-denied behavior.

Engine relevance is derived from Roamium and Surfari receive/send behavior, not
from protobuf comment groupings. A message is engine-relevant when an existing
engine receives it in dispatch, sends it through callbacks or IPC, or must
explicitly ignore/respond because Ghostboard can route it to an engine.

## Status Values

- `Missing`: Girlbat has no implemented or explicitly unsupported behavior yet.
- `Partial (headless)`: Girlbat has verified headless protocol behavior, but
  visible rendering, real-GUI integration, or full Roamium/Surfari-equivalent
  semantics remain incomplete.
- `Partial (headless lifecycle)`: Girlbat has verified owner-thread tab
  lifecycle behavior for the protocol message, but visible browser readiness,
  richer browser state, input/render teardown, or final committed browser
  semantics remain incomplete.
- `Partial (runtime lifecycle)`: Girlbat has verified owner-thread lifecycle
  behavior plus normal-page BrowserReady or structural-presentation runtime
  evidence, but visual correctness, release-install behavior, and broader
  navigation parity remain incomplete.
- `Partial (viewport-only)`: Girlbat receives the protocol message and applies
  the requested headless viewport dimensions through the Ladybird ABI, but
  screen geometry, scale, input mapping, and post-resize visual correctness
  remain incomplete.
- `Partial (color-scheme ABI)`: Girlbat receives the protocol message and calls
  Ladybird's preferred-color-scheme API, but visual/readback proof of
  `prefers-color-scheme` and full dark-mode semantics remain incomplete.
- `Partial (visibility-state ABI)`: Girlbat receives the protocol message and
  calls Ladybird's visibility-state API, but active-page throttling,
  browser-visible page-lifecycle proof, and visual/readback evidence remain
  incomplete.
- `Partial (state tracked)`: Girlbat records and reports deterministic
  owner-thread state for the protocol message, but no browser-engine effect is
  proven yet.
- `Partial (input queued)`: Girlbat receives the protocol message, routes it on
  the owner thread, and queues a bounded event through Ladybird's input ABI, but
  browser-visible input effects and full parity for omitted fields remain
  incomplete.
- `Partial (title callback)`: Girlbat emits title changes observed through
  Ladybird's title callback for deterministic normal page loads, but late
  post-load titles, intermediate coalesced titles, redirects, PDFs, crash pages,
  and history restore title parity remain incomplete.
- `Partial (console callback)`: Girlbat emits console messages observed through
  Ladybird's console callback for deterministic normal page loads, but full
  console parity for source locations, stack traces, object previews, grouping,
  tables, and every console method remains incomplete.
- `Partial (cursor callback)`: Girlbat emits cursor changes observed through
  Ladybird's cursor callback for deterministic hover over a link, mapped to the
  narrow Chromium-style cursor numbers Ghostboard currently consumes, but full
  cursor-shape parity and custom cursor support remain incomplete.
- `Partial (target-url callback)`: Girlbat emits target URL changes observed
  through Ladybird's link hover/unhover callbacks for deterministic hover over a
  link, but full status-area parity across complex hit-testing, frames,
  redirects, and rapid hover sequences remains incomplete.
- `Partial (dialog callback)`: Girlbat emits JavaScript dialog requests observed
  through Ladybird's alert/confirm/prompt callbacks for a deterministic prompt,
  but beforeunload parity and complete browser dialog UI behavior remain
  incomplete.
- `Partial (dialog reply)`: Girlbat receives JavaScript dialog replies,
  validates active request IDs, and routes prompt replies back to Ladybird so
  page JavaScript resumes, but complete alert/confirm/beforeunload UI parity
  remains incomplete.
- `Partial (crash callback)`: Girlbat can surface Ladybird WebContent crash
  callbacks as `RendererCrashed` events with conservative status/code/reload
  semantics, but deterministic engine-path delivery currently depends on the
  owner thread already pumping and full crash recovery UX remains incomplete.
- `Not engine-relevant`: Girlbat should not receive or send this message; the
  reason is documented in the behavior column.
- `Implemented`: future status for verified Girlbat behavior.
- `Unsupported`: explicit unsupported/capability-denied behavior, or an
  intentionally unused legacy engine path where sending the message would be a
  protocol regression.

Receive-side lifecycle rows remain `Missing` until full engine-relevant
semantics are implemented. A later experiment may split receive-side and
send-side coverage more finely if this conservative under-claim becomes
confusing.

Issue 884 Experiment 12 proved that the Ladybird C ABI can reach and export a
presentable `Gfx::SharedImageBuffer`. Experiments 13-18 proved that ID-only
protobuf is insufficient, that the render side channel can carry the OS surface
authority, and that Ghostboard can import a deterministic test surface.
Experiment 19 added the first metadata-only `RenderSurface` protocol seam.
Experiment 20 proves one real Ladybird-backed frame attachment can be exported
by Girlbat, imported over the side channel, and matched to nonzero
`RenderSurface.attachment_id` metadata. Experiment 21 structurally presents that
matched IOSurface through Ghostboard's AppKit overlay path. Experiment 35 proves
the same structural presentation path from a normal local HTTP page launched
through `web --browser girlbat` in Debug Ghostboard. Screenshot or
readback-backed visual correctness and continuous per-tab frame delivery remain
incomplete.
